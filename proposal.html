改善企画書

PFCバランス計算表を、
毎日つづけられる道具にする

サーバー代ゼロ・無料の範囲のまま、いちばんの不満である「食品が探しにくい」を解消し、日々の入力の手間を減らすための改善計画。入力はmacOS、栄養指導の先生への提示はiPad(またはPDF/紙)という使い方を前提にする。

作成日:2026年7月8日 対象:index.html / 栄養成分表_.csv リポジトリ:research-river/nutritional-log

1. 現状の構成

いま動いているものの整理

現在の計算表は index.html 1ファイルだけで動く静的アプリで、サーバーもデータベースも使いません。データはすべてブラウザの localStorage に保存されます。この「無料で完結する」構成は今後も変えません。

食品データ栄養成分表_.csv(8列・約40食品)を手動でアップロードして読み込み。読み込み後は localStorage に保存され、次回以降の再アップロードは不要
食事の登録朝食・昼食・夕食・間食の4区分。プルダウンで食品を選び、分量(1/3〜5)を選んで追加
集計日別のカレンダー管理。エネルギー・P・F・C・食物繊維・食塩の合計と、目標値に対するPFCバランスバー
バックアップJSON書き出し/読み込み、提出用CSV書き出し
利用環境入力はmacOSのブラウザ。スマホは使わない(モバイル回線なし)。栄養指導の先生にはiPadで見せられると理想的

2. 課題

優先度の高い順

最優先
食品が探しにくい 全食品が1本のプルダウンにフラットに並ぶため、目的の食品にたどり着くまで全リストを目視で探す必要がある。食品が増えるほど悪化する。
最優先
CSVに分類(カテゴリ)がない 現在は「□,,,,,,,」という区切り行で見た目上グループ分けしているが、アプリはこれを栄養ゼロの食品として誤って取り込んでしまう(プルダウンに「□」が並ぶ原因)。
中
毎日ほぼ同じ食事なのに、毎回選び直している 「昨日と同じ」「いつもの朝食セット」を一発で登録する手段がない。
中
CSVの更新時に手動アップロードが必要 食品を追加するたびに、CSVを編集 → アプリでファイル選択、という2段階の作業になる。
中
栄養指導の先生に見せる手段が確立していない 記録は入力したMacのブラウザ内にしか保存されないため、iPadでそのまま見ることはできない。現状は提出用CSV書き出しがあるだけで、見せ方の段取りが決まっていない。
低
糖質の列が使われていない CSVの「糖質(g)」は読み込まれているが、画面のどこにも表示・集計されていない。

3. 改善案

3つのフェーズに分けて、効果の大きいものから。すべて無料・静的ファイルのままで実現できます

Phase 1 — 食品を「探す」から「打てば出る」へ

体感がいちばん変わる改善。これだけ先にやる価値があります。

  • インクリメンタル検索:プルダウンの代わりに検索ボックスを置き、「さば」と打てば鯖の食品だけが絞り込まれて出るようにする。
  • CSVにカテゴリ列を追加:先頭に「カテゴリ」列を1列足し(例:チキン/魚/主食/乳製品…)、検索結果や一覧をカテゴリ見出し付きで表示する。「□」区切り行は廃止でき、ゴミデータ問題も同時に解決。
  • よく使う食品を先頭に:追加した回数を localStorage に記録し、使用頻度の高い食品を検索前の初期表示に出す。毎日食べるものはワンタップ圏内になる。

Phase 2 — 毎日の入力を1分以内に

「昨日と同じ」を最短にする。

  • 前日コピー:選択中の日付に、前日(または任意の日)の食事をまるごとコピーするボタン。
  • マイセット登録:「いつもの朝食」など定番の組み合わせをセットとして保存し、一発で追加。
  • CSVの自動読み込み:Netlify上で動かすなら fetch('栄養成分表_.csv') で同じサイト内のCSVを自動で読めるため、手動アップロードを廃止できる(CSVを更新して push するだけで全端末に反映)。
  • 糖質の表示・集計:読み込み済みの糖質列を、食物繊維・食塩と同様に合計欄へ追加。

Phase 3 — 栄養指導の先生に「見せる」を整える

入力はMacで完結させ、iPadは見せる専用と割り切る。

  • レポート画面+PDF書き出し:期間(1週間・1ヶ月)を指定すると、日ごとの食事内容とPFC・食物繊維・食塩の推移を印刷向けレイアウトで表示する画面を追加する。Macの印刷ダイアログから「PDFとして保存」し、AirDropでiPadに送っておけば、診察室に回線がなくてもその場で見せられる。紙に印刷しても同じものが使える。
  • iPadでの直接閲覧(Wi-Fiがある場合の補助手段):NetlifyのURLをiPadのSafariで開き、MacからAirDropで送ったJSONバックアップを読み込めば、アプリそのままの画面(カレンダー・PFCバー)で見せることもできる。ただし回線に依存するため、基本はPDFとする。

先頭に「カテゴリ」列を追加します。区切り行(□,,,,,,,)と空行は削除します。

5. Netlifyへの公開手順(備忘録)

忘れたときはここを読む

まず現状を確認する

  1. app.netlify.com にログインする(GitHubアカウントでログインしていた可能性が高い)。
  2. 「Sites」の一覧に nutritional-log 系のサイトが残っているか確認する。あればサイト名のURL(〇〇.netlify.app)をクリックして、今も表示されるか見る。

サイトが残っていた場合(GitHub連携が生きている)

何も設定し直す必要はありません。手元の変更をコミットして push するだけで、数十秒後に自動で公開されます。

cd ~/Desktop/nutritional-log
git add -A
git commit -m "Update"
git push origin main

サイトが消えていた/連携が切れていた場合

  1. Netlifyで「Add new site」→「Import an existing project」→「GitHub」を選ぶ。
  2. リポジトリ一覧から research-river/nutritional-log を選ぶ。
  3. 設定はすべてそのままでよい:Build command は空欄、Publish directory はルート(空欄または「.」)。ビルド不要の静的サイトなので何も入れない。
  4. 「Deploy」を押す。以後は push するたびに自動デプロイされる。
いちばん簡単な方法(Git不要): Netlify Drop のページに、nutritional-log フォルダをドラッグ&ドロップするだけでも公開できます。ただしこの方法は push しても更新されないので、恒久運用にはGitHub連携がおすすめです。
公開範囲について: Netlifyに置いたファイルはURLを知っていれば誰でも見られます。robots.txt で検索エンジンには載らないよう設定済みですが、食事記録データ自体は各ブラウザの localStorage にしか保存されないため、サイト経由で他人に見られることはありません。なお、無料の代替として GitHub Pages(リポジトリの Settings → Pages)もあります。

6. まとめ

おすすめの進め方

まず Phase 1 だけ実施することを推奨します。CSVにカテゴリ列を足し、プルダウンを検索ボックスに置き換える——この2つで「食品を探すのに一苦労」という最大の不満が解消され、ついでに「□」のゴミデータ問題も消えます。工数は半日〜1日程度、費用はゼロです。

その後、実際に使ってみて「毎日の入力がまだ面倒だ」と感じたら Phase 2(前日コピー・マイセット・CSV自動読み込み)へ。栄養指導の予定が近いなら、Phase 3 のレポート画面+PDF書き出しを先に作る順番でもかまいません(他のフェーズと独立して作れます)。いずれもMacだけで完結し、iPadには見せる直前にAirDropでPDFを送るだけです。
